בסיסי נתונים עם Claude Code

PostgreSQL MCP מאפשר ל-Claude לחקור schemas, לכתוב queries, ולנהל מיגרציות — ישירות בשיחה, עם הבנה מלאה של מבנה הנתונים שלכם.

⏱ 12 דקות קריאה רמה: בינוני

הגדרת PostgreSQL MCP

שרת ה-MCP של PostgreSQL מתחבר לבסיס הנתונים שלכם ומאפשר ל-Claude להריץ queries בזמן אמת.

json — claude_desktop_config.json 
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://localhost:5432/mydb"
      ]
    }
  }
}
אבטחה קריטית לעולם אל תחברו את ה-MCP ל-production database ישירות. השתמשו ב-read-only user לסביבת פיתוח, ובסביבת production — חיברו רק דרך VPN עם הרשאות מינימליות.

חקירת Schema

אחרי החיבור, Claude יכול לחקור את כל המבנה של בסיס הנתונים ולהשתמש בו כהקשר לכתיבת קוד:

שיחה עם claude 
explore the database schema and describe all tables,
their relationships, and any missing indexes you notice

Claude יחזיר סיכום מפורט של הטבלאות, foreign keys, indexes קיימים, ויציע שיפורים אם יש.

מיגרציות סכמה

מיגרציות הן הדרך הבטוחה לשנות schema בצורה מבוקרת. Claude יכתוב אותן עם up ו-down בכל מיגרציה:

  1. תיאור השינוי הרצוי 
    add a "subscription_tier" column to the users table.
values: 'free', 'pro', 'enterprise'. default: 'free'.
not null. add an index on this column.
  2. Claude יכתוב את המיגרציה 
    -- migrations/20260515_add_subscription_tier.sql
-- Up
ALTER TABLE users
  ADD COLUMN subscription_tier VARCHAR(20)
    NOT NULL DEFAULT 'free'
    CHECK (subscription_tier IN ('free', 'pro', 'enterprise'));

CREATE INDEX idx_users_subscription_tier
  ON users(subscription_tier);

-- Down
DROP INDEX IF EXISTS idx_users_subscription_tier;
ALTER TABLE users DROP COLUMN subscription_tier;
  3. בדיקה לפני הרצה

    בקשו מ-Claude לנתח השפעות לפני שמריצים:

    what's the impact of this migration on existing data and queries?
will it lock the table? estimate the time for a table with 1M rows.

דפוסי Query בטוחים

Claude מודע לסכנות נפוצות ב-SQL ויכתוב קוד שמגן מפניהן:

שיחה עם claude 
write a function to get user orders with pagination.
use parameterized queries, avoid N+1, add query timeout of 5 seconds
python — תוצאה 
async def get_user_orders(
    user_id: int,
    page: int = 1,
    page_size: int = 20,
    conn: asyncpg.Connection = None
) -> list[dict]:
    offset = (page - 1) * page_size
    query = """
        SELECT
            o.id, o.created_at, o.total_amount, o.status,
            json_agg(json_build_object(
                'product_id', oi.product_id,
                'quantity', oi.quantity,
                'price', oi.unit_price
            )) AS items
        FROM orders o
        JOIN order_items oi ON oi.order_id = o.id
        WHERE o.user_id = $1
        GROUP BY o.id
        ORDER BY o.created_at DESC
        LIMIT $2 OFFSET $3
    """
    return await conn.fetch(query, user_id, page_size, offset)

ניתוח ביצועים

Claude יכול לנתח queries איטיות ולהציע אופטימיזציות:

שיחה עם claude 
this query takes 3 seconds on the orders table (50M rows):
SELECT * FROM orders WHERE status = 'pending' AND created_at > NOW() - INTERVAL '7 days'
run EXPLAIN ANALYZE and suggest indexes
טיפ בקשו מ-Claude לכתוב את כל ה-queries עם EXPLAIN ANALYZE בסביבת פיתוח לפני שמעלים ל-production. זה חוסך הפתעות.
